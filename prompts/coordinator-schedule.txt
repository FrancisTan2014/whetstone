/every 5m Run one whetstone coordinator tick.

Use docs/LOCAL_AGENT_WORKFLOW.md as the workflow spec.

Before choosing work:
- Set GitHub CLI auth for this terminal: GH_CONFIG_DIR should be the personal config for FrancisTan2014.
- Run `.\\scripts\\cleanup-agent-locks.cmd` before reading local status or lock state.
- Read .agent-status.local.json. If it does not exist, create it from docs/agent-status.example.json.
- Read PRODUCT.md and GUIDELINES.md.
- If coordinator.paused is true, update coordinator.lastResult to paused and do not invoke any worker until the pause is cleared.

Remote status sync:
- Acquire `.agent-locks/status-sync.lock` before syncing.
- If the lock exists and is not stale, skip sync and use the previous local snapshot.
- If the lock is stale (older than 10 minutes), remove it once and retry.
- Run `git fetch origin --prune`.
- Snapshot open issues with labels, title, body, state, and URL into remote.issues.
- Snapshot open PRs with labels, title, head SHA, review markers/comments when available, check status, mergeability/conflict state when available, linked issue, and URL into remote.pullRequests.
- Update remote.lastSyncStartedAt, remote.lastSyncCompletedAt, remote.lastSyncResult, and updatedAt.
- Release `.agent-locks/status-sync.lock`.
- If after the sync attempt remote.lastSyncCompletedAt is still missing or older than 15 minutes, update coordinator.lastResult to remote_status_stale and do not invoke developer/reviewer this tick.

Scheduling decision:
1. Run `.\\scripts\\cleanup-agent-locks.cmd`.
2. If coordinator.paused is true, update coordinator.lastResult to paused and stop until the next tick.
3. If developer.failureCount is 3 or more, developer.currentIssue is set, and developer.currentPr is not set, invoke `.\\scripts\\abandon-developer-attempt.cmd` and stop. This drops the failed local worktree/branch, requeues the issue, clears pause/backoff, and lets the next tick retry cleanly.
4. If `.agent-locks/worker.lock` exists, read its `pid.txt`, `role.txt`, and `startedAt.txt`.
   - If the recorded PID is still a running `copilot` or `node` process and the lock is not older than 12 hours, update coordinator.lastResult to worker_busy and stop until the next tick.
   - If the recorded PID is missing/dead, has been reused, or the lock is older than 12 hours, remove `.agent-locks/worker.lock` once and continue.
5. If `.agent-locks/developer-claim.lock` or `.agent-locks/reviewer-work.lock` exists and is not stale, update coordinator.lastResult to worker_busy and stop until the next tick.
6. If there is no active worker lock but developer has recorded unfinished work:
   - This applies when developer.state is `failed`, or when developer.state is `implementing` / `in_progress` with developer.worktree or developer.branch set and no PR was opened.
   - For `implementing` / `in_progress` without a worker, update developer.state to `failed`, developer.resumeReason to `worker_missing_after_incomplete_run`, developer.lastResult to `process_incomplete_no_worker`, developer.lastFailureAt to now, and developer.nextRetryAfter to now if it is missing.
   - If developer.nextRetryAfter exists and is in the future, update coordinator.lastResult to developer_backoff and stop.
   - Otherwise invoke `.\scripts\start-developer.cmd` so the developer can resume the recorded worktree/branch, and stop.
7. If any PR in remote.pullRequests is labeled changes-requested:
   - If developer.nextRetryAfter exists and is in the future, update coordinator.lastResult to developer_backoff and stop.
   - Otherwise invoke `.\\scripts\\start-developer.cmd` and stop.
8. If any PR in remote.pullRequests is labeled needs-review or review-approved:
   - If reviewer.nextRetryAfter exists and is in the future, update coordinator.lastResult to reviewer_backoff and stop.
   - Otherwise invoke `.\\scripts\\start-reviewer.cmd` and stop.
9. If there is no waiting PR, find open ready-for-dev issues in remote.issues, sorted by lowest issue number.
10. Skip issues labeled in-progress.
11. If an issue declares `Depends on: #N`, skip it until every dependency issue is closed in the local snapshot.
12. If a dependency-ready issue exists:
   - If developer.nextRetryAfter exists and is in the future, update coordinator.lastResult to developer_backoff and stop.
   - Otherwise invoke `.\\scripts\\start-developer.cmd` and stop.
13. If no work exists, update coordinator.lastResult to idle and stop until the next tick.

Each developer/reviewer invocation must be a one-shot session. Do not start a recurring developer/reviewer schedule.
