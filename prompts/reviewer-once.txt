Crash resilience and durable outcome (apply this throughout the tick):
- A reviewer tick is a short-lived process that can be interrupted or crash at any time. Do the analysis synchronously in this process (foreground subagent or inline); never launch a background or detached agent and then exit.
- Reach a durable outcome before exiting: a posted GitHub review carrying the `reviewer-run-reviewed: <head-sha>` marker plus the correct label, or — for an already-approved PR — a recorded merge-gate decision (merge, or a clear reason not to).
- Order outcome steps safely: post the review first, then update labels, then merge. Do not change labels or write the marker until the review is complete, so an interruption never leaves partial label changes.
- The `reviewer-run-reviewed: <head-sha>` marker is the dedupe key. Skip a PR already reviewed at its current head SHA (unless it is review-approved and only needs a merge-gate recheck); re-review only when the head SHA changed.

Process at most one unit of reviewer work:
1. Acquire `.agent-locks/reviewer-work.lock` before selecting a PR.
2. First find one open non-draft PR in remote.pullRequests labeled review-approved and check whether merge gates now pass.
3. If no merge candidate exists, find one open non-draft PR in remote.pullRequests that needs review, preferring label needs-review.
4. Skip PRs labeled changes-requested until developer pushes fixes.
5. Skip a PR if .agent-status.local.json or PR comments show reviewer-run-reviewed for the same head SHA, unless it is labeled review-approved and only needs merge-gate checking.
6. If no review or merge work exists, update reviewer.lastResult to idle, release the reviewer work lock, and stop.
